## 1. Оновлення та зміни в архітектурі (Change Log)

Під час реалізації MVP (Minimum Viable Product) було внесено кілька ключових змін до початкової архітектури для покращення гнучкості та дотримання принципу Separation of Concerns.

| Компонент | Було | Стало | Причина зміни |
|---|---|---|---|
| VFSContext | Атрибут used_size | Метод get_used_space() | Динамічне обчислення ваги дерева надійніше за статичний лічильник. |
| VFSContext | --- | is_initialized(), has_enough_space() | Інкапсуляція перевірок стану системи всередині контексту. |
| Exec Engine | Приймав ICommand | Приймає raw_line: str | Енджин став справжнім оркестратором: він сам ініціює парсинг рядка. |
| Exec Engine | Атрибут SecurityEngine | Винесено в логіку перевірки | Спрощення взаємодії: перевірка прав відбувається безпосередньо перед запуском команди. |
| Formatter | Метод apply_colors | Вбудовано в render | Зменшення кількості дрібних методів; форматування кольору — це частина рендерингу. |

## 2. Діаграма компонентів та потік даних

Система побудована за принципом низької зв'язності. Ключову роль посередника відіграє ```main.py``` (Контролер), який координує роботу парсера та енжина.

### Напрямок потоку даних:
```Користувач/Скрипт``` -> ```Main (Controller)``` -> ```Input Parser``` -> ```Об'єкт Command (повертається в Main)``` -> ```Execution Engine``` -> ```Security Check``` -> ```Command.execute(Context)``` -> ```Output Formatter```.

## 3. Опис ключових класів

### 3.1. VFSContext (Стан системи)

Цей клас є "джерелом істини" для всієї програми.
- Методи:
    - ```get_used_space()```: Рекурсивно обчислює поточний обсяг зайнятої пам'яті.
    - ```has_enough_space(size)```: Перевіряє можливість запису нових даних.
    - ```is_initialized()```: Гарантує, що операції не проводяться без виконаної команди mkfs.

### 3.2. Execution Engine (Оркестратор)
Відповідає виключно за "життєвий цикл" виконання команди.

- Метод execute(command, raw_line): Отримує вже готовий об'єкт. Його обов'язки обмежені:
    1. Запитати SecurityEngine, чи дозволена дія.
    2. Викликати виконання команди.
    3. Передати результат у OutputFormatter.

### 3.3. Input Parser (Текстовий аналізатор)
Чистий транслятор текстових команд у програмні об'єкти.

- Принцип роботи: Використовує Dispatch Table (словник фабрик) для створення команд. Це усуває потребу в громіздких конструкціях ```if-elif``` та дозволяє легко додавати нові команди або аліаси ```(cls/clear)```.

### 3.4. Output Formatter (Модуль представлення)
Єдине місце в системі, де дозволено використання print та ANSI-кодів кольорів.

- Приймає ```CommandResult``` та перетворює його на людиночитаний вигляд (таблиці для ```ls```, текст для ```cat```, кольорові статуси ```[OK]``` або ```[ERROR]```).

## 4. Обґрунтування структур даних

1. N-арне дерево (Composite Pattern): Дозволяє системі однаково обробляти файли та директорії через інтерфейс INode. Це ідеально відповідає ієрархічній природі файлових систем.

2. Хеш-таблиці (Python Dicts):
- У Directory: Для миттєвого доступу до файлів за ім'ям (O(1)).
- У Parser: Для швидкого мапінгу рядків на класи команд.

3. Бітові маски (Bitmasks): Використовуються для зберігання прав доступу (наприклад, 755). Це дозволяє виконувати перевірки безпеки за допомогою швидких побітових операцій AND, що імітує реальну логіку прав доступу в Unix.

4. Командний патерн (Command Pattern): Кожна дія є об'єктом. Це дозволяє легко реалізувати логування, скасування дій або пакетне виконання команд зі скриптів без зміни ядра системи.
